# Phase 10.6 ‚Äî MCP elicitation (real `onElicitation` translation)

> Generated by super-planner. Source: `roadmap.md` (phase 10.6).
> Last updated: 2026-07-14 after implementation (super-implementer).

**Status:** implemented

## Goal

Replace the Phase-7 `onElicitation` cancel stub with a real translation so that
when an MCP server (the host's own in-process server, or a third-party server
surfaced by the Phase 16/17 disk scan) calls `elicit/create`, the request is
surfaced to the user instead of being silently auto-cancelled. Elicitation
routes through the **structured user-input** channel that `AskUserQuestion`
already uses (`ClaudeAgentSession.requestUserInput` ‚Üí `ActionType.ChatInputRequested`
‚Üí `respondToUserInputRequest`), **not** the `pending_confirmation` permission
gate (design locked in [CONTEXT.md:632](./CONTEXT.md)). The user's answer is
delivered back to the server as an `ElicitResult`
(`{ action: 'accept' | 'decline' | 'cancel'; content? }`).

## Scope

**In scope**

- A pure-projection module `claudeElicitation.ts` (sibling to
  `claudeInteractiveTools.ts`): JSON-Schema ‚Üí `ChatInputQuestion[]` and
  workbench answers ‚Üí `ElicitResult.content`, plus a result mapper and a
  `cancelResult()` helper. Unit-testable without an agent.
- A bridge (SDK-callback orchestration) that looks up the session, parks on
  `requestUserInput`, observes the SDK per-request `signal`, and maps the
  response back to an `ElicitResult`.
- Wiring the handler exactly like `canUseTool`: `IBuildOptionsInput` field ‚Üí
  `Options.onElicitation` projection in `buildOptions`; `IMaterializeContext`
  field; `_makeOnElicitation` closure in `ClaudeAgent`; removal of the
  `logElicitation` param and the inline stub.
- `form` mode (schema ‚Üí questions) and `url` mode (question-less request
  carrying the URL).
- Replacing Phase-7 Test 18's stub assertion with the real translation; new
  unit tests for the projection module.

**Out of scope**

- Handling `SDKElicitationCompleteMessage` (`elicitation_complete`) in the
  message router. `onElicitation` awaits the returned promise and the renderer
  resolves URL prompts via the existing `ChatInputCompleted` path; the
  out-of-band completion notification is not required for this phase (see
  Decisions). If a later phase needs true out-of-band URL completion, it owns
  that router work.
- Subagent-scoped routing of elicitations (`parentToolCallId`). Elicitation
  carries no `agentID`, so requests route to the top-level session only.
- Any change to the `IAgent` public surface, the renderer's chat-input widget,
  or the protocol `ChatInputRequest` shape (all already support `url` +
  `questions`).

## Approach

Mirror the established Claude split: a **pure** `claudeElicitation.ts` owns
schema‚Üîquestions and answers‚Üîcontent projection (modeled on
`codexElicitationMapper.ts`, adapted for the Claude SDK's **untyped**
`requestedSchema: Record<string, unknown>`), and a **bridge**
`claudeElicitationBridge.ts` owns session lookup, the `requestUserInput` park,
SDK `signal` abort wiring, and `ElicitResult` mapping (modeled on
`claudeCanUseTool.ts`). The bridge is wired through the SDK options the same way
`canUseTool` is: a new `IBuildOptionsInput.onElicitation` field is projected
onto `Options.onElicitation` in `buildOptions`, threaded via
`IMaterializeContext`, and produced by a `ClaudeAgent._makeOnElicitation`
closure at every materialize/rematerialize call site. The `logElicitation` param
and the inline `{ action: 'cancel' }` stub are deleted.

## Decisions

Resolved during council synthesis (strong 3/3 or 2/3 consensus). Grilling may add more.

- **Route through `requestUserInput`, not `pending_confirmation`.** Unanimous;
  design locked in [CONTEXT.md:632](./CONTEXT.md). Elicitation is structured
  user input, not a permission decision.
- **Separate bridge file from the pure module.** Mirror the existing split
  (`claudeCanUseTool.ts` bridge vs `claudeInteractiveTools.ts` pure). 2/3
  council + both in-repo precedents (codex, canUseTool). File name
  `claudeElicitationBridge.ts` (pure stays `claudeElicitation.ts`).
- **Fresh `generateUuid()` for the request id.** The SDK gives no `tool_use_id`
  for elicitation. Uuid guarantees global uniqueness ‚Äî required because agent
  response dispatch scans all live sessions
  ([claudeAgent.ts:1895](./claudeAgent.ts)). The abort listener captures the id
  by closure. Matches codex ([codexAgent.ts:1364](./../codex/codexAgent.ts)).
- **`SDKElicitationCompleteMessage` is out of scope.** Unanimous. `onElicitation`
  awaits the returned promise for both `form` and `url` mode; the host parks on
  `requestUserInput` and the renderer resolves via the existing
  `ChatInputCompleted` path. The `elicitation_complete` notification stays in the
  router's "known unhandled" bucket ([CONTEXT.md:1096](./CONTEXT.md)). **NB: this
  narrows the roadmap's "route it through the message router" bullet ‚Äî confirmed
  unnecessary because the promise-await model already resolves URL prompts.**
- **`parentToolCallId` omitted.** Elicitation has no `agentID`; requests route to
  the top-level session. Subagent-scoped elicitation is out of scope.
- **`decline` is reserved for explicit user Decline; everything else is
  `cancel`.** Follow Copilot's semantic ([copilotAgentSession.ts:3784](./../copilot/copilotAgentSession.ts)):
  a no-session lookup miss and a mid-park teardown both return
  `{ action: 'cancel' }`; only an explicit user Decline returns
  `{ action: 'decline' }`. This keeps one meaning for `decline` ("the user
  refused") and reports "nobody to ask" honestly as a cancel ‚Äî chosen over
  Codex's no-session‚Üí`decline` split ([codexAgent.ts:1362](./../codex/codexAgent.ts)).
  Practically: the pure module needs only a single `cancel` helper (no separate
  `declined`/`cancelled` pair), and the bridge's no-session early return and the
  teardown-driven `respondToUserInputRequest(Cancel)` both land on `cancel`.
- **No "unattended" auto-cancel branch.** Claude has no autopilot/Mode UI
  (`resolveSessionConfig` suppresses it, [claudeAgent.ts:1731](./claudeAgent.ts)),
  and sibling Codex does no auto-cancel ‚Äî it parks and lets teardown unwind. The
  handler always surfaces the prompt; parked requests unwind on dispose via
  `_pendingUserInputs.denyAll({ Cancel })`
  ([claudeAgentSession.ts:731/1087](./claudeAgentSession.ts)). This narrows the
  roadmap's "auto-cancel when unattended" bullet ‚Äî no headless Claude path
  exists today, so the branch is dead code and is omitted. A future headless
  phase owns re-adding it.

- **Permission-display metadata (`title`/`displayName`/`description`) is
  ignored.** Those SDK fields target elicitation-driven *permission* prompts,
  but Phase 10.6 routes through the user-input channel; both Codex and Copilot
  use `message` only. The bridge surfaces `request.message` as
  `ChatInputRequest.message` and drops the rest. A future permission-styled
  elicitation phase can thread them.

## Risks

- **Untyped `requestedSchema` (`Record<string, unknown>`).** Unlike codex's
  generated `McpElicitationPrimitiveSchema` and copilot's typed schema, the
  Claude SDK schema is untyped, so a malformed third-party schema could throw
  mid-projection. **Mitigation:** narrow defensively (guard every property
  access with `typeof`/`hasKey`); on any parse failure fall back to a
  question-less request (message only) so the user can still accept/decline,
  never a crash. Cover with a "malformed schema" unit test.
- **URL string ‚Üí `URI` mismatch.** `ChatInputRequest.url` is a `URI`
  ([state.ts:368](./../common/state/protocol/channels-chat/state.ts)); the SDK
  gives `url: string` ([sdk.d.ts ElicitationRequest]). **Mitigation:**
  `URI.parse` in the url-mode branch; guard against a missing/empty url.
- **`buildOptions` signature churn.** Removing `logElicitation` ripples to
  `claudeSdkOptions.test.ts` and both session build sites. **Mitigation:** step 8
  updates the test; `npm run typecheck-client` catches every caller.
- **Content coercion for numeric/array/select fields.** Answers arrive as
  `ChatInputAnswer` value kinds; coercion to the schema's declared type must
  match what the MCP server expects. **Mitigation:** port codex/copilot coercion
  (`Math.trunc` for integers, `Number.isFinite` guards, freeform-merge for
  arrays) and snapshot-test round-trips.

### Review-driven fixes (PR #325908)

Copilot's review surfaced real correctness gaps against the workbench consumer (`agentHostSessionHandler.ts`); all confirmed and fixed:

- **Schema-aware answer coercion.** The consumer renders number/integer/boolean questions as *text* inputs and returns text answers, so `"3"`/`"false"` must be coerced back to `3`/`false`. `elicitationAnswerToValue` now takes the field and coerces per `type` (ported from copilot's `elicitationAnswerToFieldValue`); uncoercible values are dropped.
- **`allowFreeformInput: false` on strict enum/select questions.** The consumer defaults an omitted `allowFreeformInput` to `true`, which would let a user submit an out-of-schema value that the translator then drops. MCP enums are strict, so the projection now disables free-form input on single/multi-select.
- **Question-less requests cancel instead of falsely accepting.** A `url`-mode request missing its URL, or a form with no representable fields, would become the consumer's injected required text question and then resolve as `accept`. The bridge now cancels a request with neither a URL nor questions.
- **Prototype-pollution safety.** Untrusted field names (`__proto__`, `constructor`, ‚Ä¶) are read with `Object.hasOwn` and content is built via `Object.fromEntries`, so such a name never reads an inherited member (crash) or mutates the prototype (silent drop).
- **Doc accuracy.** The drift-guard docstring no longer overclaims (it catches incompatible reshapes of covered fields, not purely additive new variants); the roadmap no longer lists out-of-scope `SDKElicitationCompleteMessage` router work.
- **`IElicitationField` is derived from a base-layer runtime validator and pinned to the MCP SDK's `PrimitiveSchemaDefinition`.** The final design uses `vs/base/common/validation.ts` (the pure "zod-lite") to build `vElicitationField`, runtime-validates each schema field (dropping malformed ones ‚Äî e.g. a non-array `enum` that would otherwise reach `.map` and throw), and *derives* the field type via `ValidatorType<typeof vElicitationField>` (no hand-rolled interface). A **type-only** import of `PrimitiveSchemaDefinition` plus the compile-time guard `_assertElicitationFieldCoversSchema` (verified non-vacuous and drift-catching) pins the derived type to the SDK's authoritative union. The base validator is used instead of the SDK's own zod schema because a *runtime* `@modelcontextprotocol/sdk` import doesn't resolve in the unit-test renderer (all SDK runtime access goes through `IClaudeAgentSdkService`); a `safeParse` prototype was rejected for exactly that reason. Net vs. the original hand-rolled interface: real runtime validation (fixes a latent throw), zero duplicated type, and spec-drift protection.
